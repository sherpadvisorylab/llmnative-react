# Scaffold output

This page documents the structure generated by:

```bash
npx @llmnative/react create
```

For CI or non-interactive environments:

```bash
npx @llmnative/react create --yes --name=my-app --provider=mock --ai-provider=openai --proxy-provider=viteDevProxy --theme=flat --template=crm
```

---

## Interactive prompts

The scaffold asks for:

| Prompt | Options | Effect |
|---|---|---|
| Project name | any string | Sets folder and `package.json` name |
| Data provider | `firebase` / `supabase` / `mock` / `custom` | Wires the default `data` service |
| AI provider | `none` / `openai` / `gemini` / `anthropic` / `mistral` | Wires the default `ai` service and relevant API key prompt |
| Proxy provider | `none` / `viteDevProxy` / `expressProxy` | Wires the default `proxy` service and generates the matching relay implementation |
| Icon provider | `lucide` / `phosphor` | Sets `iconProvider` in `conf/app.ts` |
| Theme | `default` / `flat` / `cyber` | Sets the built-in theme |
| App template | `blank` / `crm` / `admin` / `inventory` / `project` | Copies starter pages and mock data |

## Generated structure

### With `viteDevProxy`

```text
my-app/
  index.html
  vite.config.ts
  dev/
    llmnativeProxy.ts
  package.json
  .env
  src/
    index.tsx
    conf/
      app.ts
      menu.ts
    layouts/
    sections/
    pages/
    data/
    styles/
```

### With `expressProxy`

```text
my-app/
  index.html
  vite.config.ts
  server/
    llmnativeProxy.ts
  package.json
  .env
  src/
    index.tsx
    conf/
      app.ts
      menu.ts
    ...
```

### With `none`

No proxy relay file is generated.

## `conf/app.ts`

`conf/app.ts` is the central wiring layer for the scaffolded app.

This is where the scaffold maps env into:

- `aiConfig`
- `providers`
- `services`

The generated file now treats proxy as a real service slot:

```ts
const selectedProxyProvider = env.VITE_PROXY_PROVIDER ?? 'none';
const proxyDriver = selectedProxyProvider !== 'none'
  ? selectedProxyProvider
  : undefined;

export const providers: AppProvidersConfig = {
  proxy: {
    enabled: env.VITE_PROXY_ENABLED === 'true',
    route: env.VITE_PROXY_ROUTE ?? '/api/llmnative/proxy',
  },
  services: {
    data: dataDriver,
    auth: 'googleAuth',
    ...(aiDriver ? { ai: aiDriver } : {}),
    ...(proxyDriver ? { proxy: proxyDriver } : {}),
  },
};
```

Pages and UI components should consume hooks or framework widgets, not instantiate providers directly.

## Generated env

Scaffolded apps include:

```env
VITE_PROVIDER=mock
VITE_AI_PROVIDER=none
VITE_PROXY_PROVIDER=none
VITE_PROXY_ENABLED=false
VITE_PROXY_ROUTE=/api/llmnative/proxy
VITE_ICON_PROVIDER=lucide
VITE_THEME=default
```

The proxy env is intentionally split into:

- `VITE_PROXY_PROVIDER`
- `VITE_PROXY_ENABLED`
- `VITE_PROXY_ROUTE`

This mirrors the framework service model:

- `services.proxy`
- `providers.proxy`

## Generated relay implementations

### `dev/llmnativeProxy.ts`

Generated only when the scaffold selects `viteDevProxy`.

It mounts a Vite middleware on:

```text
/api/llmnative/proxy
```

and is activated in `vite.config.ts` only when:

```env
VITE_PROXY_ENABLED=true
VITE_PROXY_PROVIDER=viteDevProxy
```

### `server/llmnativeProxy.ts`

Generated only when the scaffold selects `expressProxy`.

It exports an Express-style registration helper that can be mounted by the application server.

---

## Why the proxy lives in the app

The library owns:

- `proxyFetch(...)`
- the proxy provider contract
- the `proxy` service slot

The application owns:

- the actual same-origin route
- the runtime where that route executes

So the scaffold creates the relay implementation in the project itself, not inside the browser runtime of `@llmnative/react`.
